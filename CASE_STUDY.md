# Discipline Dashboard — RPG-Based Career & Discipline Tracking System

## 1. Project Overview

Discipline Dashboard is a personal full-stack productivity and career tracking dashboard. It is designed to help users measure daily execution, career progress, and long-term discipline through a structured RPG-inspired system.

The project combines habit tracking, gamification, task management, and analytics into one dashboard. It turns daily actions and strategic career work into measurable XP, levels, and progress signals.

## 2. Problem Statement

Long-term career growth is difficult to measure because progress often happens slowly and across many small actions.

Daily discipline can disappear without visual feedback. Strategic goals are also often disconnected from daily execution, making it hard to understand whether short-term habits are contributing to long-term outcomes.

Traditional to-do apps usually focus on task completion, but they do not show career compounding, XP history, discipline streaks, or progress analytics. This creates a gap between productivity tracking and career development tracking.

## 3. Solution

Discipline Dashboard solves this by combining daily discipline tracking with strategic career task management.

The solution includes:

- A 180-day discipline grid
- Daily check-ins
- XP and level system
- Strategic task manager
- XP activity audit trail
- Analytics summary cards
- Secure user-specific data using Supabase Auth and Row Level Security

The result is a dashboard that makes progress visible, measurable, and motivating.

## 4. Target User

The target user is a developer, data analyst, or career switcher who wants to track daily execution and long-term growth.

This user needs a system that connects daily discipline with strategic portfolio-building, technical skill development, and career transition goals.

## 5. Technical Architecture

- Frontend: Next.js, TypeScript, Tailwind CSS
- Auth: Supabase Auth
- Database: Supabase PostgreSQL
- Security: Row Level Security
- Deployment: Vercel + Supabase
- Optional future backend: Render.com

The frontend is designed to be Vercel-ready, while Supabase provides authentication, database storage, and security policies.

## 6. Database Design

The database is organized around four core tables:

- `profiles`: Stores user profile data, total XP, current level, and the discipline campaign start date.
- `daily_checkins`: Stores daily completion records for the 180-day discipline grid.
- `xp_events`: Stores an audit trail of XP activity generated by daily check-ins and strategic task completion.
- `strategic_tasks`: Stores career, portfolio, and long-term project tasks with priority, status, category, and XP reward values.

This structure separates user identity, daily execution, XP history, and strategic work into clear database responsibilities.

## 7. Security Model

Row Level Security is enabled for user-owned tables.

Each table is scoped by either `user_id` or `auth.uid()`, ensuring users can only access their own records. This supports a secure multi-user architecture from the beginning.

The `xp_events` table is append-oriented. Normal users should be able to insert and read their own XP events, but should not update or delete XP history. This preserves auditability and prevents accidental or intentional manipulation of XP records.

## 8. Gamification Logic

The dashboard uses a simple XP model:

- Daily check-in = 100 XP
- Strategic task XP reward is configurable
- Level = `floor(total_xp / 1000) + 1`

Dynamic level titles provide motivational feedback as the user progresses. Instead of only showing a number, the dashboard communicates identity progression through RPG-style titles.

## 9. Analytics Layer

The dashboard includes an analytics summary layer with:

- Completion rate
- XP event count
- Active task count
- Completed task count
- Recent XP Activity feed

This supports Data Analytics / BI portfolio value because it demonstrates event tracking, metric calculation, dashboard design, and user-specific reporting. The data model can later be extended into charts, trend analysis, and external reporting tools such as Power BI.

## 11. Key Engineering Decisions

Supabase was chosen because it provides free-tier PostgreSQL, authentication, and Row Level Security in one platform.

Next.js was chosen for Vercel deployment, frontend performance, and future full-stack flexibility.

XP events were separated from profile totals for auditability. The profile stores fast dashboard totals, while `xp_events` preserves the historical record of how XP was earned.

Row Level Security was chosen to support secure multi-user architecture. Each user can only access their own profile, check-ins, XP events, and strategic tasks.

The project was developed incrementally: first the UI foundation, then schema planning, then authentication, then core data flows, then analytics and task controls. This reduced complexity and made each layer easier to test.

## 12. Challenges and Solutions

Email confirmation during development can affect the auth flow. The solution is to handle loading and error states clearly and configure Supabase Auth settings appropriately during local testing.

Duplicate daily XP was prevented by checking whether today already has a completed daily check-in before inserting another XP event.

Duplicate task XP was prevented by checking task status before completion. If a task is already completed, the app does not insert another XP event or add XP again.

Profile totals are kept synced with XP events by updating `profiles.total_xp` and `profiles.current_level` after successful XP-generating actions.

RLS required careful insert policies so authenticated users can create only their own records while still preventing access to other users' data.
